# 多接口处理大量数据,取消超时提醒,前后端数据分批次返回再调用

1.代码的编写以及业务的逻辑是没有问题的,在测试系统测试也是没有超时提示的,只是在发布到正式系统时,
由于正式系统的nginx有默认的响应时间设置,而当前的接口需要处理大量数据以及调用某H客户相关接口两个,某HIS客户相关接口两个,所以导致接口运行时间较慢
async与await
1
2
3
	B2B-PO这边出现了一个问题,就是我写了一个接口,发布到SMS的测试系统,测试是没有任何问题,发布到正式之后,发现接口的响应时间比在测试的响应时间长很多,测试系统基本两分钟四十秒就能跑完,正式系统得跑六分钟,这个接口需要处理大量数据,并且调用某H客户相关接口两个,某HIS客户相关接口两个,导致接口响应时间较慢,超过了正式系统的nginx设置的默认响应时长,前端会报一个504的网关响应超时的错误,不过数据还是会处理完入库再回显,就对用户可能不是很友好,当时考虑要不要把这个响应时间延长到10分钟,这样代码的逻辑就不需要更改了,问了一下Basic相关运维人员,以及网关信云相关运维人员,得到的回复是延长这个响应时间比较麻烦,因为网站本身是允许外网访问的,改这个响应时间的话,需要四层一块修改这个配置才行,这样开发周期还得加上流程审批的时间,原定的交付日期就在眼前了,所以没有采用这种方式,换了第二种方式,将数据处理的接口,按照处理的逻辑划分为多个接口,依次调用执行
1
async handleGetNewList() {
        let res = await getNormalShippedList()
        let resp = await getRidOfNormalShipped(res.data)
        let respo = await getRidOfSAP(resp.data)
        let respon = await getRidOfAPS(respo.data)
        if (respon.data.flag) {
          this.$message({
            type: 'success',
            message: 'PO数据同步成功，请查看'
          })
          if (this.multipleSelection.length > 0) {
            findNewListByPo(this.multipleSelection)
              .then(response => {
                this.tableData = response.data.data
                this.pagination.total = response.data.total
                this.$nextTick(() => {
                  this.$refs.table.doLayout()
                })
              })
          } else {
            this.findPage()
          }
        } else {
          this.$message.error(response.data.message)
        }
      },

// 获取最新Normal在途
export function getNormalShippedList() {
  return request({
    url: '/api/admin/poLineConditions/getNormalShippedList',
    method: 'post',
    timeout: 600000
  })
}

// 去已发运
export function getRidOfNormalShipped(obj) {
  return request({
    url: '/api/admin/poLineConditions/getRidOfNormalShipped',
    method: 'post',
    timeout: 600000,
    data: obj
  })
}

// 去库存
export function getRidOfSAP(obj) {
  return request({
    url: '/api/admin/poLineConditions/getRidOfSAP',
    method: 'post',
    timeout: 600000,
    data: obj
  })
}

// 去排产
export function getRidOfAPS(obj) {
  return request({
    url: '/api/admin/poLineConditions/getRidOfAPS',
    method: 'post',
    timeout: 600000,
    data: obj
  })
}

# 库表实体类设计

1.在开发过程中,会有不断地需求增加,界面展示的字段,一加再加,这样就会产生一个问题,每次都需要扒代码,找到位置获取数据,更改库表,更改实体类,存入对象mapper配置文件,需要修改的地方太多了,开发进度缓慢。
A:创建表跟实体类的时候多创建10个字段,命名为：库_表_udf1---库_表_udf10，之后再添加新的字段,直接搜索替换就可以了。
