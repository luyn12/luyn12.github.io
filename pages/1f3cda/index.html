# IOC

简介

Spring是一款轻量级的full-stack（全栈）框架。

Spring是一个平台，可以轻松的集成其他各种优秀的技术。（灵魂框架）

Spring的核心基于Core Container（核心容器），主要进行Bean的管理。

Spring的两大核心：IOC和AOP

IOC

控制反转

控制：对我们程序中所需资源（JavaBean对象）的创建和管理权的控制。

new关键字会导致我们的类与类之间紧耦合。

反转：把对象的创建权和管理权从我们自己掌控，反转到spring框架掌控。

IOC：通过spring工厂+配置文件的方式，把类与类之间的耦合尽可能削减掉。spring创建的对象，会存储在spring内置的IOC容器中，使用的时候从容器中获取这些对象即可。这个容器就是上面所说核心容器。

创建bean

  1. 使用无参构造创建对象

    ​ 使用bean标签，通过id指定唯一名称，class属性指定全类名。

<bean id="userService" class="com.kdplm.service.impl.UserServiceImpl"/>
相当于：  userService = new UserServiceImpl()
1
2

scope属性：

​ singleton（默认值）：单例，在spring容器中仅创建和维护一个对象。每次获取都使用同一个实例。单例对象在spring容器加载时就会创建，而且会存入IOC容器中。当容器关闭，单例对象都会被销毁掉。

​ prototype：多例。当我们需要获取一个对象时，每次都会创建一个新的实例返回。多例对象不会存入IOC容器，该对象不受spring管理。多例对象仅会在不使用时被垃圾回收器回收。

  1. 使用静态工厂创建对象

    class属性指定工厂的全类名，factory-method用于指定工厂中的静态工厂方法。spring会调用工厂中的这个方法，把返回值作为bean存入IOC容器中

    指定工厂的全路径之后，在指定

    <bean id="userService4" class="com.kdplm.service.UserServiceFactory" factory-method="getService">
相当于：  userService4 = UserServiceFactory.getService()

    <bean id="factoryBean" class="com.kdplm.service.UserServiceFactory2"/>
相当于：  factoryBean = new UserServiceFactory2();

<bean id="userService5" factory-bean="factoryBean" factory-method="getService" />
相当于：  userService5 = factoryBean.getService()

依赖注入（DI）

在IOC容器中管理的对象，如果它内部也依赖了其他的资源，并且我们想要给这些资源赋值。例如：Service中依赖了DAO对象，想要创建service存入IOC容器的时候把DAO的值也准备好，此时就需要spring给我们提供这些资源并把资源放到对应的位置。这个过程叫做依赖注入。

依赖注入的双方——也就是被注入对象和注入对象。都需要被spring管理。

  1. set方法注入（常用方式）

    本质就是调用bean的setter方法，把值传进去进行注入。

    分两步走！！

    ​ 首先你得确保你有要注入的变量，并且这个变量有set方法且私有

    ​ 然后注入对象和被注入对象都要被spring管理

    ​ 将要注入的资源声明为bean

    ​

    ​

    使用property标签，写到bean标签的内部。用于指定set注入。

    ​ property的name属性用于指定bean的属性名。（bean的属性名仅和setter方法的名称相关。）

    ​ value属性用于指定普通值。基本数据类型和字符串。

    ​ ref属性用于指定引用类型，指定的是spring容器中的另外一个JavaBean的id值

  2. 有参构造注入

    使用constructor-arg标签，可以给构造指定参数

    name属性：指定构造方法的参数的名称

    value或者ref：用于指定实际值

    <constructor-arg name="" value=""/>
    1

  3. 使用有参工厂初始化对象

    有参工厂方法传参方式和有参构造的传参方式一致。

在spring中引入properties文件

  1. 通过context名称空间中的property-placeholder引入properties文件

  2. location属性用于指定文件名称。在spring中，可以通过classpath或者classpath*前缀表示文件路径的获取方式。

    ​ classpath：从当前项目的类路径中获取文件

    ​ classpath* ：从当前项目所有类加载器加载的资源类路径中获取文件。包括jar包 中引入的配置文件

  3. property-placeholder标签在一个项目中只能出现一次，多次加载会覆盖。如果要加载多个文件，用逗号把文件名隔开。

  4. 默认情况下，property-placeholder会优先加载环境变量中的值。然后才加载properties文件。所以如果环境变量中有冲突的key，则取值会失败。

    一般情况下，properties文件中的key起名要不容易冲突。例如：jdbc.username

    也可以让property-placeholder不加载环境变量 。system-properties-mode="NEVER"

  5. 在配置中，使用 ${key} 的形式使用属性值就可以了。

<context:property-placeholder system-properties-mode="NEVER" location="classpath:*.properties"/>
1

在spring配置文件中使用import标签可以引入其他配置。

<import resource="classpath:applicationContext-user.xml"/>

@Order

按照数字进行排序，数字越小越先加载。

# AOP

面向对象编程：经常出现方法和方法的耦合。

面向切面编程：用于解决方法之间的耦合，能够在编译阶段完全解除方法和方法之间的耦合。

Spring中的aop采用了动态代理技术，在运行期对某些业务方法进行代码的插入。

核心概念

Target 目标对象，最终要去增强的对象就是目标对象。是增强前的对象

Proxy 代理对象，最终增强之后的对象。该对象是spring动态生成的。

JoinPoint 连接点，Target中所有的方法都叫连接点。是有资格被增强的方法。

PointCut 切入点（切点），在所有连接点中，最终被增强的连接点就是切入点。

Advice 通知，是我们想要给PointCut加入的额外功能的方法。

Aspect 切面，指的是通知和切入点的结合方式。

Weaving 织入，根据切面的描述，把通知的代码放到规定的位置和切点结合，之后一起执行。这个过程叫做织入。在spring中，是通过动态代理的方式实现的织入。织入的结果是产生了一个代理对象。

目标对象我们自己编写，通知类及通知方法也是自己编写。

通过配置的方式，告诉spring目标对象中哪些是切点。

通过配置，告诉spring哪些是通知。

通过配置，告诉spring切面的信息，也就是通知如何和切点结合。

spring会根据以上配置，自动进行动态代理，完成通知的织入，产生一个代理对象替代原有的目标对象进行执行。

配置方式

<!-- 配置目标对象到ioc容器中 -->
<bean id="userService" class="com.kdplm.service.impl.UserServiceImpl"/>
<!-- 配置通知类对象到ioc容器中 -->
<bean id="myAdvice" class="com.kdplm.aop.AOPAdvice"/>

<!--配置AOP-->
<aop:config>
    <!--配置切入点，使用切点表达式指定-->
    <aop:pointcut id="pt" expression="execution(* *..*(..))"/>
    <!--配置切面（切入点与通知的关系）-->
    <aop:aspect ref="myAdvice">
        <!--配置具体的切入点对应通知中哪个操作方法  before就是通知类型，代表通知方法在切点之前执行-->
        <aop:before method="function" pointcut-ref="pt"/>
    </aop:aspect>
</aop:config>
 <!-- 
1.将目标对象加载到IOC容器中
2.将增加的公共方法的对象加载到IOC容器中
3.配置切入点,哪个包下面的哪个方法要进行增强,execution(public User com.kdplm.service.UserService.findById(*,int))
4.将增强的公共方法,与目标方法形成切面,配置 function befor 前先运行 myAdvice
5.织入自动动态生成代理对象(增强后的对象)
-->

切点表达式尽量编写的足够精确，防止包含溢出。

通知类型

before：在切点方法之前执行

after-returning：在正常返回数据时执行

after-throwing：如果切点中有异常，则执行

after：无论切点是否抛异常，都会执行

常用的通知配置方式

<aop:before method="before" pointcut-ref="pt"/>
<aop:after-returning method="afterReturing" pointcut-ref="pt"/>
<aop:after-throwing method="afterThrowing" pointcut-ref="pt"/>
<aop:after method="after" pointcut-ref="pt"/>

around：环绕通知，可以替代上面的4种通知类型。一旦配置环绕通知，就无需再配置上面的通知了。环绕通知是使用代码的方式自己完成通知和切点的结合。

<aop:config>
   <aop:pointcut id="pt" expression="execution(* *..*(..))"/>
   <aop:aspect ref="myAdvice">
       <aop:around method="around" pointcut-ref="pt"/>
   </aop:aspect>
</aop:config>

🔴代码示例如下：

public Object around(🔴ProceedingJoinPoint pjp) {

    Object ret = null;
    try {
        System.out.println("around before...");
        //对原始方法的调用
        // 获取切点执行时传入的所有实参
        Object[] args = pjp.getArgs();
        //执行切点方法，传入实参，接收返回值
        ret = pjp.proceed(args);
        System.out.println("around after-returning...");
    } catch (Throwable throwable) {
        //System.out.println("around...exception...."+throwable.getMessage());
        System.out.println("around after-throwing...");
    } finally {
        System.out.println("around after...");
    }
    // 环绕通知的返回值最后就是代理对象调用方法时的返回值
    return ret;
}

🔴注：环绕通知有比上面4个通知更强的增强操作。由于环绕通知中有明确的切点调用，所以可以对参数和返回值也进行增强。

在通知方法上，可以声明JoinPoint类型的参数，用于获取切点调用的实际参数。

Spring中AOP的动态代理技术

默认情况下，Spring优先使用JDK的动态代理，优先使用的条件就是目标对象必须实现接口。如果没有接口，Spring底层会使用CGLIB作为动态代理的技术来完成织入。CGLIB是基于父子继承的方式实现的，也就是说最后生成的代理对象是目标对象的子类对象。使用CGLIB的条件为目标对象的类不能是最终类（final修饰的类）。

使用JDK的动态代理，生成的代理对象包是固定的，跟目标对象的包不一致。后面我们使用一些特定的扫包的框架就会出错。

我们可以强制使用CGLIB的方式进行动态代理，因为CGLIB生成的代理类对象和目标对象在相同的包下。

强制使用CGLIB

在aop配置中，使用 proxy-target-class 为true，则强制使用CGLIB。

<aop:config proxy-target-class="true">
1

注解支持的配置中，使用下面的配置开启CGLIB

<aop:aspectj-autoproxy proxy-target-class="true"/>
1

纯注解配置中，如果要强制开启CGLIB代理，使用下面的参数设置

@EnableAspectJAutoProxy(proxyTargetClass = true)
1

注解开发

@Aspect 标注在通知类上，表示该通知类中需要配置切面

@Pointcut 切入点配置，用切点表达式表示

@Around 环绕通知对应的注解

​ 由于在注解配置中，使用@Before @After @AfterReturning @AfterThrowing这几个注解，执行的顺序无法和预期保持一致，所以推荐使用环绕通知进行配置。

纯注解

在spring的配置类上添加下面的注解即可。

@EnableAspectJAutoProxy
