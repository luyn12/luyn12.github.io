架构演变

集中式

特点：所有功能代码都在一个项目中
应用场景：访问量不太大的小项目
优点：开发、部署比较简单方便
缺点：修改代码，需要对整个项目进行打包部署，相对比较麻烦
1
2
3
4

垂直拆分

特点：根据功能把项目拆分成一个个单独的系统,系统之间不进行数据交互
应用场景：访问量稍微有点大，一个项目单独承载不了（中型项目）
优点：相对于集中来说，承载的访问量跟多；系统间相互独立可以采用不同的开发语言。
缺点：代码冗余、代码复用性比较低
1

分布式

特点：根据功能把项目拆分成一个个单独的系统,系统之间进行数据交互
应用场景：访问量稍微有点大，一个项目单独承载不了（大中型项目）
优点：相对于集中来说，承载的访问量跟多；系统间相互独立可以采用不同的开发语言;代码冗余度降低；
缺点：服务间调用关系比较杂乱

SOA（分布式+服务治理工具）

特点：根据功能把项目拆分成一个个单独的系统,系统之间进行数据交互
应用场景：访问量稍微有点大，一个项目单独承载不了（大型项目）
优点：相对于集中来说，承载的访问量跟多；系统间相互独立可以采用不同的开发语言;代码冗余度降低;有服务治理工		 具，服务间调用关系比较清晰
缺点：一个系统对外提供多种服务，调用不清晰
1

微服务（分布式+服务治理工具+粒度拆分细）

特点：根据功能把项目拆分成一个个单独的系统,每个系统提供单一服务,系统之间进行数据交互
应用场景：访问量稍微有点大，一个项目单独承载不了（特大项目）
优点：相对于集中来说，承载的访问量更多；系统间相互独立可以采用不同的开发语言;代码冗余度降低;有服务治理工		 具，服务间调用关系比较清晰；一个系统对外提供一种服务;耦合度低。
缺点：系统太多，占用服务器比较多，调用关系比较复杂，部署比较麻烦

自我保护机制

eureka自我保护机制：
	心跳包正常情况下（实际心跳包>=期望心跳包数量）：
		30 客户端每隔30发送一次心跳包给服务端（instance）
		60 每隔60秒，清理一次过期的服务（服务端）
		90 超时90秒没有发送心跳包，服务过期（instance）
	实际心跳包数量小于期望心跳包数量，开启自我保护机制，不会剔除任何服务
总结：
	自我保护机制开启，实际心跳包<期望心跳包数量（触发自我保护机制），无论过期与否，都不会剔除任务服务
	---------------------------------------------------------------------------------
	自我保护机制关闭，实际心跳包>=期望心跳包数量，服务过期，剔除服务器
	自我保护机制开启，实际心跳包>=期望心跳包数量，服务过期，剔除服务器

Ribbon

概念：SpringCloud组件， (客户端负载均衡)

作用：负载均衡、简化http客户端（RestTemplate、OkHttp、HttpClient...）远程调用

原理：拦截增强（拦截HTTP请求，根据请求中【应用名称】从【注册中心】获取服务器地址列表，再使用负载均衡算法从地址列表中获取一个服务器地址，最终发送请求到指定的服务器地址）

负载均衡分类：服务端、客户端负载均衡。区分依据，服务地址选择完成在服务端还是在客户端
举例：服务端负载均衡nginx；客户端负载均衡ribbon。
1
2
负载均衡：在集群中，根据服务器性能不同，分配对应的负载（请求数）
例：
	User-Service         1000请求
	A 8G User-Service  250  1
	B 8G User-Service  250  1
	C 16G User-Service 500  2

Hystrix

概念：SpringCloud组件，熔断器

作用：保护服务器，防止出现级联失败、雪崩效应

解决“级联失败、雪崩效应”方案:
	线程隔离:
		线程池隔离（默认，访问每个服务都有独立的线程池，服务之间互不影响）
		信号量隔离
	服务降级:当出现“超时”、“异常”时，直接返回自定义友好错误提示信息
	服务熔断:及时返回错误信息，不用线程等待（close、open、halfopen）
		在一定时间，请求数、失败率达到阀值，触发熔断，关闭服务。休眠一段时间后，服务半开，接受请求，		  再次判断“一定时间内，请求数、失败率是否达到阀值”，是则继续熔断，否服务开启。
	限流：限制请求数量

//提供方：方法级别服务降级(提供方、消费方降级)：
@RestController
@RequestMapping("/consumer")
@Slf4j
public class ConsumerController {
    @GetMapping("/{id}")
    //第一步：在要降级的方法上添加注解，指定服务降级方法

        log.error("查询用户信息失败。id：{}", id);
        return "对不起，网络太拥挤了！";
    }
}

**Gateway**

概念：SpringCloud 网关组件，程序的统一入口

作用：安全、认证、鉴权、监控（日志、性能、异常监控）、黑白名单、限流、负载均衡、缓存、路由

核心：断言-->过滤-->路由

过滤方式：pre前置过滤器（对请求增强）、post后置过滤器（对响应增强）
过滤器种类：
	全局过滤器（对所有路由都生效）:不需要配置，对所有的路由都生效
		实现“GlobalFilter”、“Ordered”接口，重写接口中方法即可
		非正常结束：return exchange.getResponse().setComplete();
		继续执行：return chain.filter(exchange);
	局部过滤器（对某些路由生效）:那个路由配置了，就对那个路由生效
		1）命名规范：XxxxGatewayFilterFactory,“GatewayFilterFactory”固定
		2）一般继承“AbstractGatewayFilterFactory”,指定泛型（参考Gateway内置的过滤器即可）
		3）在重写的apply方法中接收泛型参数、进行业务逻辑处理即可
	特殊情况：
default-filters: #默认过滤器设置，局部过滤器全局化
            - AddResponseHeader=X-Response-Foo, Bar
            - AddResponseHeader=abc-myname,kdplm
     目的：让Gateway自带的一些局部过滤器能全局生效

网关核心流程详解

1）前端浏览器发送请求，请求网关工程
   例如：http://localhost:10010/api/user/123
2）网关工程接收请求，进行”断言“
   ”断言“：对请求路径进行判断，判断是否符合网关拦截规则,如下：
   		predicates:
   			- Path = 请求路径规则
3）网关工程对符合拦截规则的请求，进行“过滤”,如下：
	“过滤”：使用“过滤器”增强请求
		filters:
			- StripPrefix=1
  	http://localhost:10010/api/user/123--->http://localhost:10010/user/123
4）网关工程对“过滤（增强）”后的请求，执行“路由”,如下：
	“路由”：请求转发（网关工程不提供服务，只是一个中转站）
		uri: lb://user-service/
	1.http://localhost:10010/user/123--->
	2.根据服务名称“user-service”去Eureka注册中心拉取服务列表，使用负载均衡算法从地址列表中选中一个服务       地址，然后替换“网关服务地址：http://localshot:10010”--->
	3.最终结果：“获取的服务地址”+“/user/123”

全局过滤器：对所有路由生效(不需要配置)

		spring:

		    gateway:
